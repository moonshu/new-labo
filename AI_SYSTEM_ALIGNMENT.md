# AI 기능 역할 정의 - 시스템 스펙 정렬 및 분석 보고서 (AS-IS vs TO-BE)

## 0. 개요 및 총평

본 문서는 신규 정의된 `AI_HANDOFF.md`의 "비개입적/신호 중심 AI 원칙"을 기존 시스템 기획 문서(`INFRA_PLAN.md`, `SYSTEM_DESIGN.md`)에 녹여낼 수 있는지 분석한 결과입니다.

**결론: 충분히 반영 가능하지만, 기존 기획되어 있던 일부 '강압적인/대화형' UI 스펙의 대대적인 삭감이 필요합니다.**

기존 인프라 및 시스템 설계는 타겟 사용자가 "자체적인 기록을 멈추었을 때 시스템이 개입하여 끌어내는(Nudge/Lock)" 형태에 가까웠습니다. 이는 `AI_HANDOFF.md`에서 1순위로 경계하는 **"사고 주체의 훼손 및 행동 강제"**와 정면으로 충돌합니다. 따라서 기존 스펙 중 일부분을 **[제거]** 하거나 **[은유적 신호로 대체]**하는 방향으로 조정해야 완벽한 프로젝트 통합이 가능합니다.

---

## 1. 주요 충돌 지점과 스펙 변경 가이드 ⚠️

### 충돌 1: 강제 회고 잠금 (Reflection Lock)
- **AS-IS (기존 설계)**: `INFRA_PLAN 4.2`, `SYSTEM_DESIGN 3.1` 
  - 3일 방치 시 화면 전체 블러 처리 후 오버레이 팝업.
  - 무언가를 작성하거나 보류 버튼을 눌러야만 시스템 접근 가능 (행동 강제).
- **AI_HANDOFF 원칙**: 알림 금지 ❌, 팝업 금지 ❌, 행동 요구 금지 ❌ ("AI는 말을 걸지 않는다.")
- **TO-BE (수정안)**:
  - 오버레이 및 화면 잠금 기능은 **완전 폐기**.
  - 대신 사이드바 공간의 `Problems` 목록이나 `The Timeline`의 특정 문제(Node) UI의 **투명도(Opacity)가 흐려지거나 붉은 계열로 색감이 서서히 변하는(Fade)** 미세한 시각적 신호로만 대체합니다. (B급 - 되돌아보기 트리거 기능)

### 충돌 2: 맥락적 넛지 (Contextual Nudge)
- **AS-IS (기존 설계)**: `SYSTEM_DESIGN 3.2`
  - 빈 캔버스를 응시하거나 오랜만에 접속하면 "이전에 [A]라고 하셨는데 연결해볼까요?" 형태의 토스트 메시지 팝업 트리거.
- **AI_HANDOFF 원칙**: 관계 이유에 대한 장문 설명 ❌, AI의 방향성 힌트 제거 (D급).
- **TO-BE (수정안)**:
  - 텍스트 형태의 토스트 알림 **완전 폐기**.
  - 대신 사용자가 캔버스에서 특정 단어를 입력중일 때, 백그라운드에서 조용히 매칭하여 **작은 🔗 (링크 아이콘)**을 띄우는 것으로 대체. 사용자가 아이콘 위로 마우스를 올릴 때만(Hover) "A를 확장한 것으로 읽힐 수 있음" 이라는 한 줄 가설만 제공. (A급 - 사고 간 관계 제안 기능)

### 충돌 3: 텍스트 무조건부 요약
- **AS-IS (기존 설계)**: `INFRA_PLAN 4.1.4`
  - AI API가 호출될 때 기본적으로 '텍스트 요약'을 수행하고 뷰어에 표시 (추정).
- **AI_HANDOFF 원칙**: AI는 사고를 자신이 정리해서는 안 됨.
- **TO-BE (수정안)**:
  - 강제 텍스트 요약 렌더링 제거. 대신 코어 로직은 철저하게 **"상태(문제축, 관계, 사유 모드) 분류 및 태깅"** 용도로만 OpenAI API를 호출합니다.

## 2. 통합 후 AI 기능별 구현 스펙 (개발 백로그 연계)

기존 `INFRA_PLAN`의 아키텍처(Next.js + Supabase + OpenAI + Zustand) 뼈대는 모두 그대로 유지하면서 상단 UI/UX 렌더링 방식만 수정됩니다.

### 🟢 A급 MVP: 필수 기능
1. **문제축 연결 보조 (Problem Binding)**
   - **기존 설계 호환성**: ⭕ 매우 높음 (`INFRA_PLAN 4.1`의 매핑 제안과 일치).
   - **구현 방향**: The Canvas 하단에 `[문제 제안 칩]` 형태로 등장. 클릭하면 연결수락. (단, "새 문제 생성"을 AI가 대신 텍스트로 만들어주는 기능은 차단하고, 그저 생성의 타이밍-신호만 줍니다.)
   
2. **사고 간 관계 제안 (Relation Suggestion)**
   - **기존 설계 호환성**: 🔼 중간 (Graph Drawer 대신 미세한 아이콘으로 표출 필요).
   - **구현 방향**: The Timeline 또는 The Canvas 내에서 대상 노드 옆에 `🔗 확장/반박` 등으로 단어 수준의 칩 디자인 적용.

### 🟡 B급 MVP: 축소 유지 기능
3. **사고 상태 표시 (Cognitive State Marker)**
   - **기존 설계 호환성**: ⭕ 매우 높음 (`INFRA_PLAN`의 status 속성과 1:1 매칭).
   - **구현 방향**: 노드별로 상태 아이콘 (❓질문, 🧪가설, ⚖️판단, ⏸보류) 하나를 구석에 작게 추가합니다. 어떠한 점수 체계 등은 구현하지 않습니다.

4. **되돌아보기 트리거 (Revisit Signal)**
   - **구현 방향**: 상기 '충돌 1'에서 명시한 대로 강제 잠금 타이머를 **투명도/색상 변조 로직**으로 CSS/Tailwind 레벨에서 해결합니다. Zustand의 `isLocked` 상태는 `staleProblemIds: []` 형태로 대체됩니다. 투명해질 뿐 클릭이나 보기를 막지 않습니다.

## 3. 요약 및 제언

현재 시스템이 지닌 인프라스트럭처(Zustand 기반 빠른 렌더링, Supabase RLS 연동, API 경로 설계)는 훌륭히 설계되어 있으나, 기존 기획에는 "효율성과 게미피케이션"이 과하게 반영되어 있었습니다.

`AI_HANDOFF.md`의 새로운 비개입 원칙을 수용함으로써, 프로젝트는 불필요하게 공격적인 UI 모듈(`ReflectionLockOverlay`, `Toast Nudge`) 개발 공수를 크게 줄일 수 있으며, 대신 프롬프트 엔진의 정확도(정확히 1개의 감정/상태만 뽑아내는 제약)에 백엔드 구현 역량을 집중할 수 있게 되었습니다.
